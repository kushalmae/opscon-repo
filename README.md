# OPSCON — Spacecraft Operations Reference

A static, fully offline, browsable reference for spacecraft operators.
Drop in your CSVs/XLSXs, run the build, get a zip you can open anywhere
without a network connection.

## What it covers

Per flight-software (FSW) version:

- **Commands** — master list, args, enums
- **Telemetry** — master list, descriptions, enums, bit fields
- **Alerts** — rule, threshold, derived (math/logic) types
- **FDIR** — failure detection and isolation responses
- **Flight Rules** — operational constraints
- **Procedures** — Python command sequences and abstract checklist procedures

Every item links to the others. Click a telemetry point and you see every
alert that watches it, every procedure that reads it, every flight rule that
constrains it. Click a command and you see every procedure that issues it.

## Local build

```bash
pip install pydantic
python -m builder.build              # produces ./dist/
```

The build produces two things in `dist/`:

- **`index.html`** — single self-contained file (~7 MB) with all data, CSS,
  and JS inlined. Double-click it from a file manager and it works. No
  server needed. This is what operators get.
- **`data/` and `assets/`** — multi-file layout for when you want to serve
  the site over HTTP (internal portal, shared drive with a webserver, etc.).
  Use `python3 -m http.server -d dist 8000` to try it locally.

Pass `--no-inline` to skip the single-file output and only produce the
multi-file layout.

Strict mode fails on unresolved cross-references (good for CI gating):

```bash
python -m builder.build --strict
```

## Adding a new FSW version

1. Create `inputs/fsw-vX.Y/` mirroring an existing version's subfolders.
2. Drop in the CSVs.
3. Run `python -m builder.build` — the new version appears in the FSW
   selector at the top of the site automatically.

## CSV schemas

See `builder/schemas/__init__.py`. Every column is typed; extra columns
fail the build. To add a column, update the schema and the resolver.

Cross-reference rules:

- Alerts reference telemetry mnemonics in `watched_telemetry` (comma-separated).
- Alerts reference FDIR IDs in `fdir_id`.
- FDIR entries reference procedure IDs in `associated_procedure`.
- Flight rules reference a single telemetry mnemonic in `related_telemetry`.
- Procedures reference command and telemetry mnemonics (comma-separated).

Unresolved references emit warnings but don't fail the build (use
`--strict` to make them fatal).

## Air-gap

The site has zero CDN dependencies. All assets — JS, CSS, fonts — are local
or system stack. The zip works on a locked-down ops console.

## CI

`.gitlab-ci.yml` runs three stages:

1. **validate** — strict build, fails on any schema or reference issue.
2. **build** — produces `dist/`.
3. **package** — zips it as a downloadable artifact.

If your runner is air-gapped, build a Docker image with `pydantic`
pre-installed and reference it in the `image:` field.
